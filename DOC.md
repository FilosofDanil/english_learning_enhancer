# DOC — описание архитектуры

## Цель

Телеграм‑бот помогает заучить пары немецкая фраза → английский перевод из [CONTENT.md](CONTENT.md).

Пользователь интерфейсом видит текст на **русском**, а промпт содержит **немецкий** текст, на который нужно перевести **английский** строго по эталону из CONTENT.

Политика сравнения ответов с эталоном зафиксирована в «алгоритмическом» виде в [HOW_TO_CHECK.md](HOW_TO_CHECK.md) (_эти файлы в репозитории не изменяются_).

## Основная схема

```mermaid
flowchart LR
    User[TelegramUser] -->|Update| Dispatcher[handlers.Dispatch]
    Dispatcher -->|command switch| Commands[/start,/test,/print,/stop]
    Dispatcher -->|free text| AnswerFlow[handlers.answerText]
    AnswerFlow --> SessionStore[(session.Memory)]
    AnswerFlow --> Checker[checker.CheckAnswer]
    Commands --> Catalog[content.Loader.ParseFile]
    AnswerFlow --> Catalog
```

## Пакеты (SOLID‑укладка)

| Пакет | Задача |
|-------|--------|
| `internal/config` | Загрузка `.env`/окружение, базовые default‑ы и валидация токена. |
| `internal/content` | Парсинг markdown‑таблицы `CONTENT.md → []Phrase` без изменения самого CONTENT. |
| `internal/checker` | Токенизация, Левенштейн, `CheckAnswer` и русскоязычный HTML‑фидбек. |
| `internal/session` | Потокобезопасное in‑memory‑хранение активной сессии на `UserID`. |
| `internal/i18n` | Русские тексты сообщений пользователю. |
| `internal/bot` | Транспорт Telegram (long‑polling), `switch`‑маршрутизация команд, ответы кусками. |
| `main.go` | Сбор зависимостей (DIP): каталог файлом, память‑хранилище, адаптер чекера. |

## Интерфейсы (DIP)

- `PhraseCatalog`: читает распарсенные фразы и сырое тело CONTENT для `/print`.
- `AnswerChecker.Check(expectedEnglish, answer)`: результат с баллами и HTML‑текстом.
- `session.Store`: `Start`, `Get`, `Remove`.

## Команды

| Команда | Поведение |
|---------|-----------|
| `/start` | Приветствие и краткая инструкция (RU). |
| `/test` | Загружает все фразы, тасует порядок, начинает квиз. Если уже был активный тест — сбрасывается с сообщением пользователю. |
| `/print` | Отправляет **весь файл** CONTENT согласованного пути без HTML‑парсера (Markdown как есть кусками). |
| `/stop` | Если тест активен — итоговый баллы (за уже ответившие строки vs максимально возможное за них, плюс `X/Y` фраз всего квиза); сессию удалить. Если нет — сообщение использовать `/test`. |

Свободный текст воспринимается только как перевод активного вопроса; без сессии — подсказка запустить `/test`.

## Скоринг

Реализован текст из псевдокода раздела `CheckAnswer` в `HOW_TO_CHECK.md`: позиционные точные совпадения → опечатки `Levenshtein ≤ 1` на месте → остальные слоты через поиск среди слов пользователя до ближайшего точного/`≤1`; лишний хвост — `redundant`.

Балл за строку берётся по правилу `ratio` среди найденных «корректных» статусов. Подробные правила см. файл `HOW_TO_CHECK.md`; при расхождениях с табличными текстовыми примечаниями здесь авторитет имеет машинное описание формул там же.

Фидбек в Telegram оформляется через **ParseMode HTML** (`<b>`, `<i>`, `<code>`, `<blockquote>`), поэтому бот дополнительно экранирует пользовательские/контентные строки там, где они вставлены как данные.

## Конфигурация `.env`

| Переменная | По умолчанию | Что задаёт |
|------------|--------------|-------------|
| `TELEGRAM_BOT_TOKEN` | — | Обязательно. Bot API ключ. |
| `CONTENT_PATH` | `CONTENT.md` | Абсолютный/относительный путь до Markdown с таблицей. |
| `POLL_TIMEOUT` | `60` | Seconds long‑polling у Telegram Updates. |
| `LOG_LEVEL` | `info` | Сейчас используется флагом `telegram.Debug` когда `debug`. |

Подробнее шаг запуска: [OUTPUT.md](OUTPUT.md).

## Расширение без ломания контента

- Дописывать строки **только** в `CONTENT.md` (табличный синтаксис Markdown не менять).
- Нужно другое хранилище сессии — замените `session.Memory` на реализацию того же интерфейса в `main.go`.
- Новый язык UI — добавьте пакеты `internal/i18n_<lang>` и переключайте по будущей настройке (сейчас только RU строки собраны в `internal/i18n/ru.go`).
